鉴权系统

这个项目的鉴权机制在当前的形态下，属于非常典型的“前端防君子，后端防不住”的初始演示版本。作为你的前端项目架构分析导师，我可以明确地告诉你：目前的鉴权机制是非常不完善的，存在严重的安全漏洞，无法直接用于真实的生产环境。

让我们像剥洋葱一样，一层层剖析现有的架构逻辑，以及它所面临的致命威胁。

一、 现有的鉴权架构是如何运转的？​

目前项目的鉴权逻辑完全依赖于前端的本地存储（LocalStorage）和客户端的路由拦截：

  1. 登录签发：用户在登录页输入账号密码后，后端的 /api/auth/login 接口只是简单地在数据库里比对密码是否一致。如果一致，就给前端返回一个包含 role 和 id 的普通 JSON 对象。
  2. 前端存储：登录页面的 handleSubmit 函数拿到这个 JSON 后，将 userRole、username 和 userId 直接以明文形式存入了浏览器的 localStorage 中。
  3. 路由守卫：当用户访问后台页面时，app/(admin)/layout.tsx 会去读取 localStorage 里的 userRole。如果发现角色是 ADMIN，就放行到管理员页面；如果是 MERCHANT，就放行到商户页面；如果没有，就踢回登录页。

二、 架构层面的致命漏洞分析​

这套机制在前后端分离的现代架构中，存在以下四个必须要解决的核心问题：

1. “伪造身份”漏洞（前端鉴权的局限）​

由于前端路由守卫完全依赖 localStorage，而 localStorage 是可以被用户随意篡改的。 攻击场景：一个普通访客，甚至不需要账号，只需打开浏览器的开发者工具（F12），在 Console 里输入 localStorage.setItem('userRole', 'ADMIN')，刷新页面后，你们前端的路由守卫就会将其视为管理员并放行。

2. “裸奔”的后端 API（零接口级鉴权）​

这是最严重的问题。前端的路由拦截仅仅是不让用户“看”到页面，但并没有阻止用户“调”接口。 攻击场景：观察管理员审核酒店的接口 /api/admin/hotel/route.ts，里面没有任何验证调用者身份的代码。任何人只要知道这个接口地址，使用 Postman 或终端发送一个携带 id 和 status: 'APPROVED' 的 PATCH 请求，就能轻易篡改数据库中任何一家酒店的审核状态。

3. 越权访问漏洞（IDOR / 水平越权）​

在商户获取自己酒店列表的接口 /api/merchant/hotel/route.ts 中，后端是通过读取 URL 参数中的 merchantId 来查询数据的（searchParams.get('merchantId')）。 攻击场景：一个商户（假设他的 ID 是 10），登录后发起的请求是 /api/merchant/hotel?merchantId=10。他只需要把 URL 里的数字改成 11，就能直接获取到竞争对手（商户 ID 为 11）的所有酒店和房型数据，因为后端完全没有校验“当前发起请求的人，到底是不是 ID 为 11 的商户”。

4. 密码明文存储风险​

登录接口中的校验逻辑 if (!user || user.password !== password) 暴露了数据库中的密码是以明文形式存储的。一旦数据库泄露，所有用户的密码将直接暴露，缺乏最基础的哈希（Hash）加密保护。

三、 架构进阶：如何打造坚不可摧的鉴权体系？​

为了让易宿酒店管理平台达到企业级的数据安全标准，我们需要将鉴权核心从“前端信任”转移到“后端校验”。

  1. 引入无状态令牌机制 (JWT) 后端在验证密码成功后，不应该只返回普通数据，而是应该使用用户的 ID 和 Role 签发一个 JSON Web Token (JWT)。这个 Token 使用后端的密钥进行了数字签名，前端无法篡改（一改签名就会失效）。
  2. 安全的状态存储 (HttpOnly Cookies) 不要把令牌存放在 localStorage 里。应该由后端通过 Set-Cookie 头，将 JWT 写入客户端的 HttpOnly Cookie 中。这样可以彻底阻断 XSS（跨站脚本攻击）窃取凭证的可能。
  3. 构建后端中间件拦截 (API Guard) 在 Next.js 的服务端层面，为 /api/admin/* 和 /api/merchant/* 建立中间件拦截。每一个请求到达实际的业务逻辑之前，必须先解析 Cookie 中的 JWT：
    • 如果没有 JWT 或已过期 -> 返回 401 拒绝访问。
    • 如果有 JWT，但访问 /admin 接口的 Token 角色不是 ADMIN -> 返回 403 越权拒绝。
  4. 消除参数依赖 (解决 IDOR) 对于商户获取自己数据的接口，绝对不能从前端传过来的 URL 参数或 Body 中读取 merchantId。必须从后端经过验证的 JWT 载荷（Payload）中提取出对应的真实用户 ID，用这个 ID 去查数据库。

你可以先评估一下这些架构升级的思路。如果你准备好重构这部分安全防线，我们可以深入探讨在 Next.js 中如何优雅地运用中间件（Middleware）或者专门的鉴权库（如 NextAuth.js）来落地这些概念。

如果直接使用localStorage 会出现水合错误（客户端与服务端渲染不一致）：如果你的登录页代码在初始渲染（Render）阶段，直接读取了浏览器的 localStorage 或 window 对象来判断页面布局（比如有没有 token），由于 Next.js 服务端渲染时没有 window，会导致服务端生成的 HTML 和客户端接管后生成的 DOM 不一致。React 会强行“推翻重绘”，产生肉眼可见的闪烁。

这两个现象在 Next.js 的工程化开发中非常典型，分别对应了前端架构中的**“水合不匹配（Hydration Mismatch）/ 样式闪烁（FOUC）”问题，以及“开发环境按需编译（Lazy Compilation）”**机制。

作为前端项目架构分析导师，我们逐一拆解这两个现象的底层逻辑，并给出根治方案。

一、 刷新 Login 页面时的“乱码闪烁”现象​

架构剖析（为什么会闪烁）： 这通常是由以下两种情况引起的：

  1. 客户端与服务端渲染不一致（水合错误）：如果你的登录页代码在初始渲染（Render）阶段，直接读取了浏览器的 localStorage 或 window 对象来判断页面布局（比如有没有 token），由于 Next.js 服务端渲染时没有 window，会导致服务端生成的 HTML 和客户端接管后生成的 DOM 不一致。React 会强行“推翻重绘”，产生肉眼可见的闪烁。
  2. UI 组件库的 CSS 按需注入延迟：如果你使用了 Ant Design 等组件库，且没有配置 App Router 的服务端 CSS 注册表（Registry），页面的 HTML 会先于组件的 CSS 结构到达浏览器。那一两秒钟，你看到的是“没有穿衣服的裸体 HTML”，等 JS 加载完把 CSS 注入进去后，页面才瞬间排版正常。

解决方案：将“登录状态拦截”前置到 Proxy 中间件 之前我们只用 proxy.ts 拦截了未登录访问后台的请求，现在我们要加上**“已登录用户访问登录页”**的拦截。让服务端直接完成重定向，连登录页的 HTML 都不下发，彻底消灭闪烁。

打开你项目根目录的 proxy.ts，在提取 token 之后加上一段逻辑：

  // 1. 梳理需要拦截的路由规则 
  const isLoginPage = pathname === '/login' // 或者是你实际的登录路由路径
  
  // ... 其他逻辑不变 ...

  // 2. 从 Cookie 中提取 Token
  const token = request.cookies.get('yisu_token')?.value

  // 👉 新增逻辑：如果已经有 token，且访问的是登录页，直接在服务端把它踢到对应的后台，不让它渲染登录页！
  if (token && isLoginPage) {
    try {
      const { payload } = await jwtVerify(token, JWT_SECRET)
      const safeRole = (payload.role as string || '').toUpperCase()
      
      if (safeRole === 'ADMIN') {
        return NextResponse.redirect(new URL('/admin', request.url))
      } else if (safeRole === 'MERCHANT') {
        return NextResponse.redirect(new URL('/merchant', request.url))
      }
    } catch (e) {
      // token 无效或过期，放行让它正常渲染登录页
    }
  }

  // 原来的未登录拦截逻辑
  if (!token && !isLoginPage) {
    return handleUnauthorized(request, '未登录，缺少身份令牌')
  }
