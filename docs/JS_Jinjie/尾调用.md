**尾调用优化**（Tail Call Optimization，简称 TCO）。

为了让你清晰地理解，我们分两步来看：先搞懂“什么是尾调用”，再看“它到底是怎么被优化的”。

---

## 一、 什么是尾调用（Tail Call）？

**尾调用**的定义非常简单：**一个函数的最后一步操作，是调用另一个函数。**

在尾调用发生后，当前的函数就没有任何额外的工作需要做了，它只需要把被调用函数的返回值，直接作为自己的返回值传出去。

### 代码对比示例

**1. 这是尾调用：**
```javascript
function f(x) {
  return g(x); // 函数 f 的最后一步仅仅是调用 g(x)，并且直接返回它的结果
}
```

**2. 这些不是尾调用：**
```javascript
// 情况一：调用后还有额外的计算
function f1(x) {
  return g(x) + 1; // 虽然 g(x) 在最后一行，但最后一步操作其实是 "+ 1"
}

// 情况二：隐式的额外操作
function f2(x) {
  let result = g(x);
  return result; // 实际上是在赋值后再返回，引擎处理时会有额外步骤
}

// 情况三：没有明确 return
function f3(x) {
  g(x); 
  // 隐式地 return undefined; 所以最后一步并不是调用 g(x)
}
```

---

## 二、 为什么要优化？（调用栈的痛点）

要理解优化，首先要知道不优化会发生什么。

在计算机执行代码时，每当调用一个函数，系统都会在内存的**调用栈（Call Stack）**中推入一个**栈帧（Stack Frame）**。栈帧用来保存这个函数的局部变量、参数以及执行到哪里的“返回地址”。

如果函数 `A` 内部调用了函数 `B`，那么在 `B` 执行完之前，`A` 的栈帧必须一直保留在内存中，因为 `B` 执行完后还要回到 `A` 继续执行后续代码。
如果这种调用嵌套非常深（比如深层递归），内存中就会堆积大量的栈帧，最终导致内存耗尽，引发著名的错误：**栈溢出（Stack Overflow）**。

---

## 三、 尾调用优化（TCO）是如何做的？

尾调用优化的核心思想是：**复用栈帧**。

既然“尾调用”是当前函数的最后一步操作，当前函数已经不需要做任何事情了，它的局部变量也不再需要了。那么，**我们为什么还要保留当前函数的栈帧呢？**

编译器或解释器在开启了尾调用优化后，会做如下处理：

1. **丢弃旧栈帧：** 当执行到尾调用时，直接弹出（或覆盖）当前函数 `A` 的栈帧。
2. **复用空间：** 将被调用的函数 `B` 的栈帧直接放在 `A` 原来的位置上。
3. **直接返回：** 让 `B` 执行完毕后，将其返回值直接返回给 `A` 的调用者，完全跳过 `A`。

### 优化的巨大收益：尾递归

尾调用优化最经典的应用场景是**尾递归（Tail Recursion）**。
普通的递归由于不断调用自身，空间复杂度是 $O(N)$，极易爆栈。但如果我们将递归改写成尾调用的形式，配合编译器的尾调用优化，空间复杂度就会降为 $O(1)$，它在底层实际上变成了一个高效的 `while` 循环，永远不会发生栈溢出。

### 尾调用的优化条件
1. **必须是严格模式：** 因为在非严格模式下，`this` 指向全局对象，而在严格模式下，`this` 是 `undefined`，这会影响到尾调用的优化。
2. **调用是函数的最后一步操作** 
3. **调用函数不能引用当前函数作用于中的变量** 
4. **尾调用的结果就是函数的返回值**

> **注意：** 并不是所有编程语言都支持尾调用优化。像 Haskell、Scheme 等函数式语言天生支持；C、C++ 在开启优化选项（如 `-O2`）时支持；而 Python、Java 默认是不支持的；JavaScript 在 ES6 标准中规定了支持，但在实际浏览器引擎（如 V8）的实现中由于调试困难等原因，支持情况存在争议或需要特定条件。
