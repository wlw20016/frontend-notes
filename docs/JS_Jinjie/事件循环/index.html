事件循环

为了把执行顺序理清楚，我们先对任务进行极简的“分类”，然后直接看它的“运转规则”。

1. 任务的三大阵营​

在 JS 中，所有的代码执行都被分为了三类：

  • 同步任务（主线程代码）：正常的变量声明、普通的函数调用、console.log 等。它们在**执行栈（Call Stack）**中立刻排队执行。

  • 微任务（Microtask - VIP 快速通道）：

  • Promise.then / catch / finally

  • MutationObserver (浏览器)

  • process.nextTick (Node.js，且优先级比 Promise 更高)

  • 宏任务（Macrotask - 普通排队通道）：

  • setTimeout / setInterval

  • DOM 事件回调（比如点击事件）

  • 网络请求（Ajax / Fetch 回调）

  • 文件 I/O (Node.js)

2. 事件循环的“绝对顺序”（黄金 5 步）​

只要 JS 引擎还在运行，它就会死死遵守下面这 5 个步骤，不断循环往复：

  1. 执行所有的同步代码：把当前执行栈里能同步跑完的代码全部跑完，直到执行栈清空。
  2. 🔴 清空微任务队列（最关键的一步）：去微任务队列里看有没有排队的任务。如果有，就挨个执行，直到微任务队列彻底为空。
  • 注意：如果在这个过程中，微任务又产生了新的微任务，新的微任务会被追加到队尾，并在当前这一轮直接被清空，绝对不会留到下一轮！
  1. UI 渲染（仅限浏览器）：浏览器会根据刷新率（通常是 60Hz，即 16.6ms 一次）评估是否需要重新渲染页面。
  2. 🟢 执行【一个】宏任务：去宏任务队列中，取出排在最前面的仅仅一个宏任务，把它推入执行栈中执行。
  • 注意：微任务是一次性清空一窝，而宏任务是一次只执行一个！
  1. 回到第 2 步：这个宏任务执行完（也就相当于新的一波同步代码执行完），立刻回头去检查微任务队列。如此往复。

3. 用一个经典例子来验证​

看下面这段代码，你可以顺着上面的黄金步骤在脑子里跑一遍：

console.log('1. 同步代码: 开始');

setTimeout(() => {
  console.log('2. 宏任务: setTimeout 执行');
  Promise.resolve().then(() => {
    console.log('3. 微任务: setTimeout 里的 Promise 执行');
  });
}, 0);

Promise.resolve().then(() => {
  console.log('4. 微任务: 第一个 Promise 执行');
});

console.log('5. 同步代码: 结束');

大脑解析引擎启动：

  1. 第一步（走同步）：打印 1。遇到 setTimeout，把它的回调扔进宏任务队列。遇到 Promise.then，把它的回调扔进微任务队列。接着打印 5。执行栈清空。
  • 此时输出：1, 5
  1. 第二步（清空微任务）：去微任务队列一看，有个 4 在排队，拿出来执行。打印 4。此时微任务队列空了。
  • 此时输出：1, 5, 4
  1. 第三步（走一个宏任务）：去宏任务队列一看，拿到了 setTimeout 的回调，放进主线程执行。打印 2。
  • 此时输出：1, 5, 4, 2
  • 注意：在执行这步时，又遇到了一个 Promise.then，把它扔进微任务队列。
  1. 第四步（回到第二步：再次清空微任务）：刚执行完一个宏任务，立刻检查微任务队列，发现里面有个 3，拿出来执行。打印 3。
  • 最终输出：1, 5, 4, 2, 3

只要记住 “同步 -> 全部微任务 -> 一个宏任务 -> 全部微任务 -> 一个宏任务...” 这个闭环，你就掌握了 JS 异步调度的真理。
