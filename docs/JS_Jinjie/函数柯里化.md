## 什么是函数柯里化
**函数柯里化（Currying）** 是函数式编程中的一种核心思想。从逻辑和架构的角度来理解，它是**将一个接受多个参数的函数，转换为一系列只接受单个参数的函数的过程**。

用数学表达来类比，就是将原本一次性接收所有参数的执行方式 $f(a, b, c)$，拆解并重构成依次接收参数的执行链 $f(a)(b)(c)$。
```js
function add(a) {
  return function (b) {
    return a + b;
  }
}
add(1)(2) // 3
```

## 柯里化面试题
需求：改写函数实现如下效果
```js
function sum(a,b,c,d,e){
    return a+b+c+d+e
}
//改写函数sum实现：参数传递到5个即可实现累加
//sum(1)(2,3)(4)(5) // 15
//sum(1,2,3)(4)(5) // 15
//等等
```
思路：保存不定长参数，每次调用都将参数累加，直到参数数量达到5个，才触发累加操作。
```js
//1.保存不定长参数
let nums = []
function sum(...args){
    nums.push(...args)
    if(nums.length >= 5){
        //截取前五个数相加
        const res = nums.slice(0,5).reduce((a, b)=> a + b)
        //清空数组，准备下一次累加
        nums = []
        return res
    }else{
        return sum
    }
}
```
优化：参数的个数可以自定义
```js
function sumMaker(len){
    let nums = []
    return function sum(...args){
    nums.push(...args)
    if(nums.length >= len){
        //截取前五个数相加
        const res = nums.slice(0,len).reduce((a, b)=> a + b)
        //清空数组，准备下一次累加
        nums = []
        return res
    }else{
        return sum
    }
}
}
```

## 函数柯里化 实际应用
为函数预制通用参数，供多次重复调用
```js
//动态生成了类型判断函数
function typeOfTest(type){
    return function(obj){
        return typeof obj ===type 
    }
}
typeOfTest('undefined')(undefined) // true
typeOfTest('string')('hello') // true
typeOfTest('number')(123) // true
```

柯里化在实际开发中最经典的两个应用场景是：**参数复用（业务解耦）** 和 **中间件架构（逻辑增强）**。

### 1. 业务级应用：API 请求的参数复用（配置预设）

在像酒店预订这样的前后端分离项目中，我们通常会有很多接口请求。这些请求往往共享相同的域名（Base URL），某些模块又共享相同的请求方法（GET/POST）。如果不使用柯里化，每次请求都要把这些参数传一遍。

通过柯里化，我们可以像“流水线”一样，一层层剥离固定参数，最终生成极其简洁的业务调用函数。

```typescript
// 1. 基础的柯里化请求封装
// 接收基础 URL -> 接收请求方法 -> 接收具体路由 -> 返回最终处理数据的异步函数
function createApi(baseUrl: string) {
  return function(method: 'GET' | 'POST' | 'PUT' | 'DELETE') {
    return function(endpoint: string) {
      // 最终执行的函数：只接收业务数据，并通过泛型 <T> 返回对应的响应类型
      return async function<T>(data?: Record<string, unknown>): Promise<T> {
        const url = `${baseUrl}${endpoint}`;
        
        // 模拟真实的 fetch 逻辑
        console.log(`[发起请求] ${method} ${url}`, data || '无参数');
        
        // 这里仅为演示，直接返回一个模拟的 Promise
        return Promise.resolve({} as T); 
      };
    };
  };
}

// --- 实际架构流转与复用 ---

// 第一层：固化业务域名，生成针对该主站的 API 实例
const yisuApi = createApi('https://api.yisu-hotel.com/v1');

// 第二层：按请求方法进行模块化拆分
const getRequest = yisuApi('GET');
const postRequest = yisuApi('POST');

// 第三层：绑定具体路由，生成极其纯粹的业务函数
const fetchHotelList = getRequest('/hotels');
const submitOrder = postRequest('/orders');
const fetchAdminDashboard = getRequest('/admin/stats');

// --- 最终的业务调用层 ---
// 此时，开发者根本不需要关心域名、方法、路由，只需要专注传入当前的业务状态！
fetchHotelList<{ hotels: string[] }>({ city: 'Chongqing', date: '2026-05-01' });
submitOrder<{ success: boolean; orderId: string }>({ hotelId: 'A102', roomId: 'R2001' });
```

**代码剖析**：柯里化利用了 JavaScript 的**闭包**特性。`fetchHotelList` 执行时，它依然“记着”外层函数传入的 `'https://api.yisu-hotel.com/v1'` 和 `'GET'`。这使得底层的通用逻辑与上层的业务逻辑实现了解耦。

### 2. 架构级应用：Redux 的中间件模式

在状态管理（如 Redux）中，中间件的核心思想是在 `dispatch(action)` 到达 `reducer` 之前，拦截并增强它（比如加日志、处理异步等）。Redux 的中间件源码就是柯里化的极致体现。

它的标准形态是连续三次单一参数接收：`store => next => action => { ... }`。

```typescript
// 定义 Redux 核心类型 (简化版)
interface Action {
  type: string;
  payload?: unknown;
}

interface Store {
  getState: () => Record<string, unknown>;
  dispatch: (action: Action) => void;
}

type NextFunction = (action: Action) => void;

// 这是一个标准的日志中间件 (Logger Middleware)
const loggerMiddleware = (store: Store) => {
  // 第 1 层：框架初始化时注入全局 Store，供内部获取状态
  return (next: NextFunction) => {
    // 第 2 层：框架组合中间件时注入 Next 函数，用于把控制权交给下一个中间件
    return (action: Action) => {
      // 第 3 层：实际运行时拦截到的业务 Action
      
      // 在这最内层的函数里，我们集齐了所有需要的上下文！
      console.log('--- 动作派发前状态 ---', store.getState());
      console.log('--- 执行动作 ---', action.type);
      
      // 将 action 放行，传递给下一个中间件或最终的 reducer
      next(action);
      
      console.log('--- 动作派发后状态 ---', store.getState());
    };
  };
};
```

**代码剖析**：为什么要拆成三层，而不是写成 `logger(store, next, action)`？
因为这三个参数**不是在同一个生命周期里准备好的**。
* `store` 是在应用刚启动（createStore）时就确定的。
* `next` 是在中间件组合（applyMiddleware）时动态绑定的。
* `action` 是在用户点击按钮，程序实际运行期间产生的。

柯里化完美地契合了这种**参数按阶段就绪**的客观规律，让底层架构可以优雅地分发上下文。
