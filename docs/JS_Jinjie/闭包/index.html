闭包

什么是闭包​

闭包 = 内层函数 + 外层函数的变量

function outer(){ //外层函数
    let a = 10;   //外层函数的变量
    function inner(){ //内层函数
        console.log(a);
    }
    return inner;
}
const innerFn = outer();
innerFn(); // 10

闭包的作用​

  1. 变量私有化，保护变量不被随意修改
  2. 延长变量的生命周期（垃圾回收机制）
  3. 会存在内存泄漏 的问题

闭包的实现原理​

要彻底理解 JavaScript 中闭包的实现原理，我们需要潜入 JavaScript 引擎的底层，了解它是如何管理内存和作用域的。

闭包的本质并不是什么魔法，而是 JavaScript 词法作用域（Lexical Scoping）规则与垃圾回收机制（Garbage Collection）相互作用的自然结果。

我们可以通过以下三个核心概念来拆解它的底层原理：

1. 词法环境 (Lexical Environment) 与 [[Environment]] 属性​

在 JavaScript 中，每一段可执行代码（如全局代码、函数）在执行时，都会创建一个执行上下文（Execution Context）。而每个执行上下文中都包含一个词法环境（Lexical Environment）。

  • 词法环境可以看作是一个用来存储变量和函数声明的“字典”（环境记录），同时它还包含一个指向外部词法环境的引用（outer reference）。 这也就是作用域链的形成
  • 关键点： 当一个函数在 JavaScript 中被创建时，引擎会给这个函数对象附加一个隐藏的内部属性 [[Environment]]。这个属性死死地记住了该函数诞生时的外部词法环境。这也就是作用域链的形成基础

2. 作用域链 (Scope Chain)​

当函数被执行时，如果它需要访问一个变量，它会按照以下顺序查找：

  1. 首先在自己的局部词法环境中找。
  2. 如果找不到，就顺着隐藏属性 [[Environment]] 记录的外部引用去外层词法环境中找。
  3. 这样一层一层往上找，直到全局词法环境。这根由外部引用串起来的链条，就是作用域链。

3. 垃圾回收 (Garbage Collection) 的“网开一面”​

JavaScript 引擎拥有自动垃圾回收机制。通常情况下，当一个函数执行完毕后，它的执行上下文会被销毁，其内部的局部变量占用的内存也会被回收。

但是，闭包打破了这个常规： 如果外层函数返回了一个内层函数，并且内层函数（通过它的 [[Environment]] 属性）依然引用着外层函数的词法环境，那么 JavaScript 的垃圾回收器就会发现：“外层函数的词法环境还有人在用，不能删！” 于是，外层函数的执行上下文虽然结束了，但它的词法环境被保留在了内存堆（Heap）中，这就是闭包能够“记住”状态的根本原因。

💡 核心原理图解（代码步步追踪）​

让我们用一段最简单的代码，从引擎的视角走一遍流程：

function outer() {
  let a = 10;
  function inner() {
    console.log(a);
  }
  return inner;
}

const myClosure = outer();
myClosure(); // 输出 10

底层执行步骤：

  1. 编译阶段：引擎看到 outer 函数声明。
  2. **执行 outer()**：
  • 创建 outer 的执行上下文，压入调用栈。
  • 创建 outer 的词法环境，里面存着变量 a = 10 和函数 inner。
  • 引擎为 inner 函数打上烙印：设置 inner.[[Environment]] = outer的词法环境。
  1. **outer() 执行完毕，返回 inner**：
  • outer 的执行上下文从调用栈中弹出。
  • 正常情况：outer 的词法环境应该被销毁。
  • 闭包情况：因为全局变量 myClosure 现在指向了 inner 函数，而 inner 的 [[Environment]] 依然紧紧抓着 outer 的词法环境。所以，a = 10 所在的这个词法环境幸存了下来，驻留在内存中。
  1. 执行 myClosure()（即执行 inner）：
  • 创建 inner 的执行上下文。
  • 遇到 console.log(a)，在 inner 自己的词法环境中找不到 a。
  • 顺着 inner.[[Environment]] 提供的线索，找到了那个被保留下来的 outer 的词法环境。
  • 成功读取到 a = 10 并打印。

总结​

简单来说，闭包的原理就是：函数自带一个隐藏属性 [[Environment]] 记录了老家（定义时的作用域）。只要这个函数还活着（被引用），它的老家就不会被垃圾回收机制拆除，里面的数据（变量）也就一直能被访问。

闭包的生命周期​

当 myClosure(); 调用结束以后，发生的事情可能会有些出乎意料，也是很多初学者容易产生误解的地方：inner 函数本身的执行上下文会被销毁，但是闭包（outer 的词法环境）依然安然无恙地存活在内存中。

让我们具体拆解一下调用结束后发生的三个核心动作：

1. inner 的执行上下文被销毁​

当你调用 myClosure()（也就是 inner()）时，JavaScript 引擎会为它创建一个临时的“执行上下文”并压入调用栈。 当这行代码执行完毕后，这个临时的执行上下文就会从调用栈中弹出并销毁。如果在 inner 函数内部声明了任何局部变量，这些局部变量占用的空间会被回收。

2. 闭包（outer 的词法环境）继续存活​

这是最关键的一点！myClosure() 执行完了，不代表闭包就消失了。

为什么？因为在全局作用域中，变量 myClosure 依然指向 inner 函数所在的内存地址。 只要 myClosure 这个变量还在，inner 函数对象就还在；只要 inner 函数对象还在，它身上的 [[Environment]] 隐藏属性就依然紧紧抓着 outer 的词法环境（里面装着 a = 10）。

垃圾回收器的逻辑是：“既然全局变量还在用这个函数，这个函数的‘老家’就得一直留着。”

如果你再次调用 myClosure()，它依然能准确无误地访问到那个被保留下来的 a = 10。

// 如果外层函数里有状态改变，这点会更加明显：
function createCounter() {
  let count = 0;
  return function() {
    count++;
    console.log(count);
  }
}

const counter = createCounter();
counter(); // 调用结束，输出 1。但 count=1 这个状态依然保留在内存里
counter(); // 再次调用，输出 2
counter(); // 再次调用，输出 3

3. 如何真正销毁这个闭包？（释放内存）​

既然调用结束不能销毁闭包，那闭包会一直占用内存吗？ 是的。如果是在全局作用域定义的 myClosure，只有当用户关闭这个网页（或退出 Node.js 进程）时，内存才会被释放。这就是为什么滥用闭包会导致内存泄漏的原因。

如果要在代码中主动释放这块内存，你必须切断引用：

const myClosure = outer();
myClosure(); // 输出 10

// 主动打断引用，释放闭包内存
myClosure = null;

当你把 myClosure 设为 null 后：

  1. inner 函数对象失去了最后一个引用，变成了“内存孤儿”。
  2. 垃圾回收器下次巡逻时，会把 inner 函数对象回收。
  3. inner 函数没了，它对 outer 词法环境的引用也就断了。
  4. 最终，装有 a = 10 的那个 outer 词法环境也被彻底销毁，内存归还给系统。

总结一下： 也就是说，只有myClosure被释放，myClosure指向的inner才会被释放，inner.[[Environment]]指向的outer才会被释放
